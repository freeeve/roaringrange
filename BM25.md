# RRSB — roaring range BM25 impact sidecar (`RRSB`, version 1)

> **Status: implemented** behind the non-default `terms` Cargo feature — out-of-core
> builder (`write_impacts` + `ImpactsAccumulator`), a range-fetchable reader
> (`ImpactIndex`), a wasm binding (`RrbIndex`), and the cross-mode rerank query path
> (`ImpactIndex::rerank` / `search_bm25`). Live on the OpenAlex demo (default-on for the
> term and hybrid-term modes). This document is the frozen byte layout the reader/writer
> agree on; the impact math lives in one place, `quantize_impact`.

An **additive** member of the roaringrange family (next to the trigram `RRS`, term `RRTI`,
facet `RRSF`, vector `RRVI`, record `RRSR`, lookup `RRIL`, sort columns `RRSC`, and the
`RRHC` boot accelerator). `RRSB` adds **BM25 lexical relevance** to the term index without
touching it: one precomputed impact byte per `(term, doc)` posting, addressed through the
**same posting bitmaps the term search already fetched**. A reader that never opts in never
fetches a byte of it; the `.rrt` stays exactly as it was.

## Why head_off keying (not ordinals)

The sidecar is keyed by each term's **posting-region `head_off`** — the byte offset of that
term's postings inside its `.rrt`, which the term dictionary already records and which is
**unique and ascending** in dictionary order. Keying on the byte layout (not a term ordinal)
means a doc's impact is found with arithmetic the term search has already done: a query term's
posting bitmap is fetched to intersect, and **`posting.rank(doc) − 1` is the index of that
doc's impact byte** within the term's impacts run. No second lookup structure, no re-derived
ordering. (The corollary: the sidecar is valid only against the exact `.rrt` it was built
from — a rebuilt `.rrt` with different `head_off`s needs a rebuilt `.rrb`. The build fails
loudly if the dictionary `head_off`s aren't ascending or a term is missing stats.)

## File layout

```
[ header        64 B                      ]
[ sparse index  sparse_count × 8 B        ]   every stride-th entry's head_off (resident)
[ entry table   term_count × 20 B         ]   (head_off, impacts_rel, df), ascending head_off
[ impacts blob  Σ df bytes                ]   one impact byte per (term, doc), dict order
```

### Header (64 B, little-endian)

| offset | size | field | notes |
|---|---|---|---|
| 0 | 4 | magic | `RRSB` |
| 4 | 2 | version | `1` |
| 6 | 2 | pad | `0` |
| 8 | 4 | `scale` (f32) | dequantization scale = `k1 + 1.0` |
| 12 | 4 | `k1` (f32) | BM25 tf-saturation (default `1.2`) |
| 16 | 4 | `b` (f32) | BM25 length-normalization (default `0.75`) |
| 20 | 4 | `avgdl` (f32) | mean document length (tokens) |
| 24 | 4 | `term_count` (u32) | dictionary terms = entry-table rows |
| 28 | 4 | `stride` (u32) | entries per resident sparse key (`512`) |
| 32 | 8 | `entries_off` (u64) | absolute offset of the entry table |
| 40 | 8 | `impacts_off` (u64) | absolute offset of the impacts blob |
| 48 | 8 | `doc_count` (u64) | corpus size — BM25's `N` |
| 56 | 8 | reserved | `0` |

`k1`/`b`/`avgdl` are recorded for provenance; the reader scores with `scale`, `df`, and
`doc_count` only.

### Sparse index

`sparse_count = ceil(term_count / stride)` u64s — the `head_off` of every `stride`-th entry,
held resident after `open` (O(vocabulary / stride), ~8 bytes per 512 terms). It bounds an
entry lookup to a single ranged read of one ~10 KB stripe.

### Entry table

`term_count` records of 20 B, in **ascending `head_off`** order:

| size | field | notes |
|---|---|---|
| 8 | `head_off` (u64) | the term's posting head offset in the `.rrt` — the key |
| 8 | `impacts_rel` (u64) | this term's impacts run start, relative to `impacts_off` |
| 4 | `df` (u32) | document frequency (docs containing the term) |

### Impacts blob

`Σ df` bytes. Each term contributes `df` bytes at `impacts_off + impacts_rel`, in the term's
**ascending doc order** — the same order as its posting — so the `i`-th byte is the impact of
the doc at posting rank `i+1`. Each byte is `quantize_impact(tf, dl, avgdl, k1, b)`, the BM25
tf component with the document-length norm folded in, scaled to `1..=255` (never `0`: presence
in the posting implies a nonzero contribution):

```
s      = tf · (k1 + 1) / (tf + k1 · (1 − b + b · dl / avgdl))
byte   = clamp(round(s · 255 / (k1 + 1)), 1, 255)
impact = byte · scale / 255            // dequantized at query time
```

## Query path

The reader **reranks** a candidate set (it does not produce candidates): the term/trigram
index supplies matching doc IDs, and `RRSB` scores them.

1. For each query term, take its posting `head_off` from the `.rrt` dictionary and look up
   `(impacts_rel, df)`: binary-search the resident sparse index to the covering stripe, fetch
   that one stripe, binary-search it for `head_off`. All query terms coalesce into **one wave**.
2. `idf = ln(1 + (N − df + 0.5) / (df + 0.5))`, with `N = doc_count`.
3. For each candidate doc present in the term's (already-fetched) posting bitmap, read its
   impact byte at `impacts_off + impacts_rel + (posting.rank(doc) − 1)`. These single-byte
   reads coalesce into **one wave** — candidates are intersection-clustered and ascending, so
   their byte offsets cluster into a few contiguous reads.
4. `score(doc) = Σ_terms idf · byte · scale / 255`; return the top-`k` (ties broken by
   ascending doc ID, i.e. descending static rank).

Because the sidecar is keyed in the shared rank-ordered doc-ID space, it reranks candidates
from **any** producer over that space — including a term *split set*: the split set returns
global doc IDs, the monolith `.rrt` + `.rrb` score them (see [SPLITSET.md](SPLITSET.md)).

## Status / conformance

- **Rust** — builder `write_impacts` + `ImpactsAccumulator` (out-of-core; the full-corpus
  484M `.rrb` is built chunked), reader `ImpactIndex` with `rerank` / `search_bm25`.
- **Browser** — `RrbIndex` (wasm), used by the demo's term/hybrid relevance toggle.
- **Go / Python** — none (Rust-only builder + Rust/wasm reader).
